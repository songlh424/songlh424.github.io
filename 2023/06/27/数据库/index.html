MySQL

MySQL执行流程是怎样的

MySQL架构共分为两层：Server层和存储引擎层。

  • Server层负责建立连接、分析和执行SQL。主要包括连接器、查询缓存、解析器、预处理器、优化器、执行器等。另外所有内置函数（如日期、时间等）和所有跨存储引擎的功能（如存储过程、触发器、视图等）都在Server层。
  • 存储引擎负责数据的存储和提取。支持InnoDB、MyISAM、Memory等多个存储引擎。

  1. 连接器：先连接MySQL服务

    空闲连接不会一直占用，由wait_timeout参数控制，默认8小时

    MySQL最大连接数由max_connections参数控制

    解决长连接占用内存的问题：1.定期断开长连接；2客户端主动重置连接mysql_reset_connection()函数

  2. 查询缓存：以key-value形式保存，key为SQL查询语句，value为SQL语句查询的结果。只要一个表由更新操作，那么这个表的缓存就会被清空。

  3. 解析SQL：

    解析器：做词法分析、语法分析，注意，表不存在或字段不存在并不是在解析器做的

  4. 执行SQL

    预处理器：检查SQL查询语句中的表或字段是否存在；将select中的 * 替换为表上所有列。

    优化器：为SQL查询语句制定一个执行计划，比如表里面有多个索引优化器会基于查询成本的考虑来决定选择哪个索引。

    执行器：判断权限。执行过程中，执行器就会和存储引擎交互，交互以记录为单位的。

    索引下推：能减少二级索引在查询时的回表操作，提高查询的效率，因为它将Server层部分负责的事情交给存储引擎去处理了。
    例如对age和reward字段建立联合索引，现有下面查询语句：
    select * from t_user where age > 20 and reward = 10000;
    联合索引遇到范围查询就会停止匹配，也就是age字段能用到联合索引，但reward字段无法利用索引。不适用索引下推，执行器与存储器执行流程是这样的：

    • Server层首先调用存储引擎接口定位到满足查询条件的第一条二级索引；
    • 存储引擎根据二级索引的B+树快速定位到这条记录后，获取主键值，然后进行回表操作，将完整记录返回给Server层；
    • Server层再判断该记录的reward是否等于10000，如果成立将其发送给客户端，否则跳过该记录；
    • 接着，继续向存储引擎要下一条记录，存储引擎再二级索引定位到记录后获取主键值再回表将完整记录返回给Server层。
    • 如此往复，直到存储引擎把表中所有记录读完。

    没有索引下推，每查询到一条二级索引记录都要回表操作，然后将记录返回给Server，接着Server再判断记录的reward是否等于10000，适用索引下推，判断的工作交给存储引擎层，过程如下：

    • Server层调用存储引擎的接口定位到满足查询条件的第一条二级索引记录
    • 存储引擎定位到二级索引后，先不执行回表操作，而是先判断以下该索引中包含的列（reward）的条件是否成立。如果条件不成立直接跳过该二级索引，如果成立则指向回表将完整记录返回给Server层
    • Server层再判断其它查询条件是否成立，如果成立则发送给客户端，否则跳过向存储引擎索要下一条记录
    • 如此往复，直到存储引擎把表中所有记录读完。

一行记录是怎么存储的

MySQL在5.0引入了Compact这种紧凑的行格式：

记录头信息：包含delete_mask表示该数据是否被删除；next_record指向下一条记录；record_type表示当前记录的类型，0表示普通记录，1表示非叶子节点记录，2表示最小记录，3表示最大记录。

row_id：没有指定主键也没有唯一约束索引，就会加上row_id隐藏字段，占用6字节。

trx_id：事务id，必需的，占用6字节

roll_pointer：该记录的上一版本指针，必须的，占用7字节。

MySQL的NULL值是怎么存放的？MySQL的Compact行格式会用NULL值列表来标记值为NULL的列，NULL值并不会存储在行格式中的真实数据部分。NULL值列表会占用1字节空间，用二进制位逆序记录列是否为null，当表中字段都是NOT NULL，就没有NULL值列表。

MySQL怎么直到varchar(n)实际占用数据的大小：MySQL的Compact行格式会用变长字段长度列表存储边长字段实际占用的数据大小。且存储字节数的顺序与真实数据存放的顺序相反。当表中字段没有变长的就没有变长字段长度列表。

varchar(n)中的n最大取值为多少？一行记录最大能存储65535字节数据，但是这包含「变长字段字节数列表所占用的字节数」和「NULL值列表所占用的字节数」。
如果一张表只有一个 varchar(n) 字段，且允许为 NULL，字符集为 ascii。varchar(n) 中 n 最大取值为 65532。计算公式：65535 - 变长字段字节数列表所占用的字节数 - NULL值列表所占用的字节数 = 65535 - 2 - 1 = 65532。

行溢出后MySQL怎么处理？如果一个数据页存不了一条记录，InnoDB 存储引擎会自动将溢出的数据存放到「溢出页」中。Compact 行格式针对行溢出的处理是这样的：当发生行溢出时，在记录的真实数据处只会保存该列的一部分数据，而把剩余的数据放在「溢出页」中，然后真实数据处用 20 字节存储指向溢出页的地址，从而可以找到剩余数据所在的页。

MySQL存储引擎MyISAM和InnoDB的区别

  1. InnoDB支持事务、支持外键、是聚集索引（数据文件和主键索引绑在一起），不保存表的具体行数，不支持全文索引，支持表、行级锁，必须有唯一索引

  2. MyISAM不支持事务、不支持外键、是非聚集索引，保存表的具体行数，支持全文索引，表格可以被压缩后进行查询，只支持表级锁。

  3. MyISAM适合读操作和插入操作为主，只有很少的更新和删除操作，并且对事务性、并发性要求不高，是Web、数据仓储等应用环境下最常使用的存储引擎。

    InnoDB：如果在并发条件下要求数据一致性，数据操作除了插入和查询外，还包括很多的更新、删除操作。除了有效降低由于删除和更新导致的锁定还可以确保事务的完整提交和回滚，对于类似计费系统或财务系统对数据准确性要求较高系统。

索引

什么是索引

索引定义就是帮助存储引擎快速获取数据的一种数据结构，索引是数据的目录。

所谓存储引擎就是如何存储数据、如何更新、如何查询数据以及如何为存储的数据建立索引等技术的实现方法。

索引的分类

  • 数据结构分类：B+ tree索引、Hash索引、Full-text索引

    InnoDB存储引擎索引的选择：

    • 如果有主键，默认会使用主键作为聚簇索引的索引键
    • 如果没有主键，则选择第一个不包含NULL值的唯一列作为聚簇索引的索引值
    • 上面两个都没有，InnoDB将自动生成一个隐式自增id列作为聚簇索引的索引值

    其它索引都属于辅助索引，也称为二级索引或非聚簇索引。创建的主键索引和二级索引默认使用B+ Tree索引。

  • 物理存储分类：聚簇索引（主键索引）、二级索引（辅助索引）

  • 字段特性分类：主键索引、唯一索引、普通索引、前缀索引

  • 字段个数分类：单列索引、联合索引

联合索引：例(a,b)联合索引

select * from table where a > 1 and b = 2：会先找到一条符合a>1的索引，再在链表向后扫描，但是在符合a>1条件的二级索引记录的范围里，b字段是无序的。只有a走联合索引，b不走。

select * from table where a >= 1 and b = 2：找到第一条a>=1的索引，然后沿着链表向后扫描，直到不符合a>=1，当二级索引记录的a字段为1时，可以通过b=2条件减少需要扫描的二级索引记录范围。也就是说从符合a=1 and b=2条件的第一条记录开始扫描，而不需要从a为1的第一条记录就开始扫描。

select * from table where a between 2 and 8 and b = 2：在mysql中，between类似 >= and <=，类似上一条，a、b字段都用到了来联合索引。

联合索引的最左匹配原则，遇到范围查询(如>、<)的时候，就会停止匹配，也就是范围查询字段可以用到联合索引，但是在范围查询字段的后面字段无法用到联合索引。注意对于>=、<=、between、like前缀匹配的访问查询，并不会停止匹配。

索引下推：可以在联合索引遍历过程中，对联合索引中包含的字段先做判断，直接过滤掉不满足条件的记录，减少回表次数。出现了Extra为Using index conditin说明了使用了索引下推的优化。

联合索引进行排序：如何通过索引提高select * from order where status = 1 order by create_time asc的查询效率——单独给status建索引，还要通过文件排序来对create_time排序。应该给status和create_time列建立联合索引，因为这样根据status筛选的数据就是按照create_time排好序。

什么时候需要/不需要创建索引

索引好处：提高查询速度

索引坏处：

  • 需要占用物理空间，数量越大占用空间越大
  • 创建索引和维护索引要耗费时间，随数据量的增加而增大
  • 会降低表的增删改的效率。

什么时候适用索引

  • 字段有唯一性限制
  • 经常用于where查询的字段
  • 经常用于group by和order by字段

什么时候不需要创建索引

  • where,group by,order by用不到的字段
  • 字段存在大量重复数据，即区分度不高的
  • 表数据太少
  • 经常更新的字段

优化索引的方法

前缀索引优化

前缀索引就是使用字符串的前几个字符建立索引

使用前缀索引是为了减小索引字段大小，可以增加一个索引页中存储的索引值。

局限性：order by无法使用前缀索引；无法把前缀索引用作覆盖索引。

覆盖索引优化

覆盖索引：在二级索引的B+ Tree就能查询到的过程就叫做覆盖索引。

主键索引最好是递增的

使用自增主键，每次插入的新数据就会按顺序添加到当前索引节点的位置，不需要移动数据，当页面写满时，就会自动开辟一个新页面。

使用非自增主键，可能需要移动数据的位置，甚至需要从一个页面复制到数据到另外一个页面，这种情况下称为页分裂。页分裂有可能造成大量的内存碎片，导致索引结构不紧凑，影响查询效率。

另外主键字段长度不要太大，因为主键长度越小，意味着二级索引的叶子节点越小，这样二级索引占用空间也越小。

索引最好设置NOT NULL

  • 索引列存在NULL会导致优化器在做索引选择的时候更加复杂，更加难以优化，因为含NULL列会使索引、索引统计和值比较都更复杂，必须进行索引统计时，count会省略值为NULL的行。
  • NULL值是一个无意义的值，但会占用物理空间，所以会带来存储空间的问题，因为InnoDB存储记录的时候，如果表中存在允许为NULL，那么行格式中至少会用1字节空间存储NULL值列表。

防止索引失效

索引失效情况：

  • 使用左或者左右模糊匹配时，如%xx或%xx%

  • 查询条件中对索引列做了计算、函数、类型转换操作

    类型转换中，MySQL会自动【字符串】转换成【数字】，即原字段是整形，SQL语句中写的是字符串，能走索引，相反则不能走索引。

  • 联合索引不按照最左匹配原则

  • where子句中，or前是索引类，or后不是也会使索引失效

通过explain获取执行计划：

参数有：

  • possible_keys字段表示可能用到的索引
  • key实际用到的索引
  • key_len索引长度
  • rows表示扫描的数据行数
  • type表示数据扫描类型

type字段常见的扫描类型执行效率从低到高的顺序：

  • all（全表扫描）
  • index（全索引扫描）
  • range（索引范围扫描）——从这一级别开始索引的作用越来越明显
  • ref（非唯一索引扫描）——有重复数据
  • eq_ref（唯一索引扫描）——使用主键或唯一索引，通常使用在多表联查中。
  • const（结果只有一条的主键或唯一索引扫描）——使用主键或唯一索引与常量进行比较
  • Extra

Extra字段

  • Using filesort：当查询语句中包含group by操作，而且无法利用索引完成操作是会选择相应的排序算法进行，甚至会通过文件排序，效率很低
  • Using temporary：使用临时表保存中间结果，MySQL在对查询结果排序时使用临时表，常见于排序order by和分组查询group by，效率低。
  • Using index：所需数据在索引即可全部获得，无需再到表中取数据，也就是使用了覆盖索引，避免回表。

InnoDB如何存储数据的

InnoDB的数据是按【数据页】为单位读写的，数据库的I/O操作的最小单位是页，InnoDB数据页的默认大小是16KB。

数据页结构图：

文件头中有两个指针将数据页以双向链表的结构管理。

数据页中的细节图：

数据页中的记录按照【主键】顺序组成单向链表，单向链表检索效率低，数据页有一个页目录。数据会被分组，规定第一个分组的记录能有1条记录，最后一个分组中的记录条数范围只能在1~8条，其余分组记录条数为4 ~ 8条之间。每个分组最后一条记录的n_owned字段记录了该分组的条数。页目录由多个槽组成的，槽相当于分组记录的索引，指向每个分组的最后一条记录。

查找时可以对槽进行二分查找。

B+树如何进行查询的

InnoDB采用了B+树作为索引，每个节点都是一个数据页

B+树特点：

  • 只有叶子节点才存放了数据，非叶子节点仅用来存放目录项作为索引。
  • 非叶子节点分为不同层次，通过分层来降低每一层的搜索量
  • 所有节点按照索引键大小排序，构成了一个双向链表，便于范围查询

非叶子节点的数据页中的记录是最小主键id和页地址，通过二分查找找到下一层的页号，直到找到叶子节点，再根据槽二分查找找到第一条满足条件的记录数据。

为什么MySQL采用B+树作为索引

磁盘读写的最小单位是扇区，大小只有512B，操作系统一次会读写多个扇区，所以**操作系统的最小读写单位是块（Block），Linux中的块大小为4KB**，也就是一次磁盘I/O操作会直接读写8个扇区。索引查找某行数据时，就需要先从磁盘读取索引到内存，再通过索引从磁盘中找到某行数据，然后读入内存。查询过程会发生多次磁盘I/O。

单点查询

B树进行单个索引值查询时，会比B+树稍快一些，但B树查询波动会比较大。

B+树非叶子节点不存放实际的记录数据，可以存放更多的索引，磁盘IO次数更少。

插入和删除效率

B+树有大量的冗余节点，使得删除一个节点时，可以直接从叶子节点中删除，甚至可以不动非叶子节点，这样删除快。

B+树的插入也是一样，插入可能存在节点的分裂，但是最多只涉及树的一条路径，而且B+树会自动平衡，不需要复杂的算法。

B+树的插入和删除效率更高。

范围查询

B+树所有叶子节点还有一个链表进行连接，对范围查找非常有帮助。

MySQL单表能存多少

非叶子节点指向其它页的数量

每个数据页大概有1KB的固定信息，剩下15K用于存数据，索引页记录的是主键与页号，主键假设Bitint（8byte），页号是固定的（4byte），一条数据12byte。

索引 15 * 1024 / 12 ~= 1280行。

叶子节点能容纳的数据行数

能放数据的空间也是15K，叶子节点存放的是真正的行数据，按一条行数据1K来算，能存15条。

  • B+树是两层，Total = 1280^1^ x 15 = 19200
  • B+树是三层，Total = 1280 ^2^ x 15 = 24576000

count(*) 和 count(1)哪个性能好？

count(*) = count(1) > count(主键字段) > count(字段)

**count(主键字段)**：有二级索引就用，没有就用聚簇索引，将读取到的记录返回给server层，读取记录中的主键值，判断不为NULL就将count变量加1.

count(1)：将读取的记录返回给server层，但不会读取记录中的任何字段值，每读取一条记录就将count加1.

*count()**：MySQL会将*转换为参数0处理，同count(1)基本一样。

**count(字段)**：采用全表扫描来计数。

由于InnoDB存储引擎支持事务，同一个时刻的多个查询，由于多版本并发控制（MVCC）原因，应该返回多少行是不确定，所以无法像MyISAM一样只维护一个row_count变量。

优化count(*)

面对一张大表用count(*)做统计所需时间是比较大的。

  • 如果不要很精准，可以使用show table status或者explain命令进行估算。
  • 额外保存计数值，将这个值保存到单独的一张计数表，插入一条记录的同时将计数表字段+1.

事务

事务的特性ACID

  • 原子性：一个事务中的所有操作要么全部完成，要么全部不完成，事务在执行过程中发生错误，会回滚到事务开始前的状态。
  • 一致性：指事务操作前和操作后，数据满足完整性约束，数据库保持一致性状态。
  • 隔离性：数据库允许多个并发事务同时对数据进行读写和修改的能力，隔离性可防止多个事务并发执行时由于交叉执行而导致数据的不一致。
  • 持久性：事务处理结束后，对数据的修改是永久的。

InnoDB通过什么来保证事务的四个特性

  • 持久性通过redo log（重做日志）来保证；
  • 原子性通过undo log（回滚日志）来保证；
  • 隔离性通过MVCC（多版本并发控制）或锁机制来保证；
  • 一致性通过持久性+原子性+隔离性来保证；

并行事务会引发的问题

脏读

如果一个事务读到了另一个未提交事务修改过的数据

不可重复读

在一个事务内多次读取同一个数据，出现前后两次读到的数据不一样的情况

幻读

解决了脏读和不可重复读

MySQL InnoDB引擎的默认隔离级别是【可重复读】，但很大程度上避免幻读现象。解决方案两种：

  • 快照读（普通select语句），通过MVCC方式解决了幻读，事务的执行过程中看到的数据一直跟这个事务启动时看到的数据是一致的。
  • 当前读（select…for update等语句），通过next-key lock（记录锁+间隙锁）方式解决了幻读，当执行select…for update语句的时候，会加上next-key lock，如果有其他事务在该锁范围内插入了一条记录，这条语句会被阻塞。

串行化

会对记录加上读写锁，在多个事务对这条记录进行读写操作时，如果发生了读写冲突时，后访问的事务必须等前一个事务执行完成才能继续执行

解决了脏读、不可重复读和幻读

Read View在MVCC如何工作

Read View有四个字段

  • m_ids：指的是创建Read View时，当前数据库中活跃事务的事务id列表，活跃事务值得是启动了还没提交的事务。
  • min_trx_id：指创建Read View时，当前数据库中活跃事务中id最小事务，即m_ids中最小值。
  • max_trx_id：创建Read View时当前数据库中应该给下一个事务的id值，即全局事务中最大的事务id值+1；
  • creator_trx_id：创建该Read View的事务的事务id

聚簇索引中两个隐藏列

  • trx_id：当一个事务对某条聚簇索引记录进行改动时，就会把该事务的事务id记录在trx_id隐藏列里；
  • roll_pointer：每次对某条聚簇索引记录进行改动时，都会把旧版本的记录写入undo日志中，然后这个隐藏列是个指针，指向每一个旧版本记录，可以通过它找到修改前的记录。

Read View

一个事务访问记录，处理能看见自己更新的记录，还有几种情况：

  • 如果记录的trx_id小于Read View的min_trx_id值，表示这条记录的版本在创建Read View前已提交，对当前事务可见
  • 如果记录的trx_id值大于等于Read View中的max_trx_id值，表示创建Read View后才启动的事务生成，对该事务不可见
  • 如果trx_id值在min_trx_id和max_trx_id之间，需要判断trx_id是否在m_ids列表中：
    • 如果在，表示生成该版本记录的活跃事务依然活跃着，对当前事务不可见
    • 如果不在，表示生成该版本记录的事务已提交，对当前事务可见

这种通过版本链来控制并发事务访问同一个记录时的行为就叫MVCC（多版本并发控制）。

读提交如何工作

读提交隔离级别是在每次读取数据时，都会生成一个新的Read View。

可重复读隔离级别出现幻读的场景

事务A先是快照读，之后事务B插入并提交，之后事务A当前读就会导致两次读的结果集不一样。

表明MySQL可重复读隔离级别并没有彻底解决幻读，只是在很大程度上避免了幻读现象的发生。针对上面的幻读，尽量在开始事务之后，马上执行select…for update这类当前读语句，它会对记录加next-key lock，从而避免其它事务插入一条新记录。

MySQL有哪些锁

全局锁

要使用全局锁，则要执行这条命令：

flush tables with read lock

执行后，整个数据库只读。对表的增删改或对表结构的更改都会被阻塞。

如果要释放执行：

unlock tables

应用场景：用于全库逻辑备份。

缺点：数据多时备份时间长，且期间只能读书不能更新数据，会造成业务停滞。

避免方式：如果数据库的事务支持可重复读的隔离级别，在备份数据库之前开启事务，会创建Read View，由于MVCC支持，备份期间仍然可以对数据进行更新操作。

备份数据库的工具是mysqldump，在使用mysqldump时附上-single_transaction参数时，就会在备份数据库之前开启事务。

但对于MyISAM这种不支持事务的引擎备份数据库时只能使用全局锁。

MySQL表级锁有：

  • 表锁
  • 元数据锁（MDL）

表锁

加表锁：

表锁会限制别的线程，也会限制本线程。

释放表锁：

尽量避免在InnoDB引擎的表使用表锁，因为表锁粒度太大，影响并发性能，InnoDB的牛逼的地方在于实现了更细粒度的行级锁。

元数据锁（MDL）

不需要我们显示使用，当我们对数据库表进行操作时会自动给这表加上MDL。

  • 对一张表进行CRUD操作时，加的是MDL读锁；
  • 对一张表做结构变更操作时，加的是MDL写锁；

MDL为了保证用户对表执行CRUD操作时，防止其他线程对这个表结构做变更。

MDL是在事务提交之后才会被释放，意味着事务执行期间MDL一直持有。

MDL写锁会导致后续申请读锁的查询操作被阻塞，因为申请MDL锁的操作会形成一个队列，队列中写锁获取优先级高于读锁，一旦出现MDL写锁等待，会阻塞后续该表的所有CRUD操作。

意向锁

当执行插入、更新、删除操作会对某些记录加独占锁，此时会先在表级别上加一个意向共享锁。当对表中某些记录加共享锁时，会现在表级别上加一个意向共享锁。

意向锁之间不发生冲突，只会和表级共享锁和表级独占锁发生冲突。

意向锁的目的是为了快速判断表里是否有记录被加锁。

AUTO-INC锁

自增主键在后续插入数据时可以不指定主键的值，数据库自动给主键赋值递增，主要通过AUTO-INC锁实现的。

是特殊的表锁，在执行完插入语句后就会立即释放，插入数据时会加一个表级的AUTO-INC锁。保证auto_increacement修饰的字段的值是连续递增的。但是会阻塞插入操作，影响性能。

在MySQL5.1.22版本后，提供了一种轻量级的锁来实现自增，给该字段赋值一个自增的值就把锁释放了，不需要等待插入整个语句执行完才释放。

当使用这个轻量级的锁同时binlog日志格式是statement，即记录的语句就是原始语句，在主从复制的场景会发生数据不一致的问题。

如果有两个session同时执行向表中插入数据，会导致其中一个session的insert的id是不连续的，而binlog是分session记录语句。当这个binlog拿到从库执行，从库按顺序执行，不会像两个session同时执行，这样导致每个session的id都是连续的，就发生了数据不一致的问题。

解决这个问题是：binlog格式设置为row，这样binlog里面记录的是主库分配的自增值，到从库执行时主库自增值是什么从库自增值就是什么。

行级锁

普通select不会加行级锁，属于快照读，通过MVCC实现一致性读，无锁的。

select可以加共享锁和独占锁

上面两条语句必须在一个事务中，事务提交了，锁就释放了。

Record Lock

记录锁，锁住一条记录，有S锁和X锁之分。

Gap Lock

间隙锁，只存在于可重复读隔离级别，为了解决可重复读隔离级别下的幻读现象。

间隙锁也分X和S，但之间是兼容的，不存在互斥关系，都是防止插入幻影记录而提出的。

Next-Key Lock

锁定一个范围并且锁定记录本身。也分X和S，但是X互斥的。

插入意向锁

一个事务要插入，需判断插入位置是否被其他事务加了间隙锁，如果有，插入就阻塞，直到拥有间隙锁的事务提交，在这个期间会生成一个插入意向锁，表明有事务想在某个区间插入新纪录，现在处于等待状态。

插入意向锁不是意向锁，是一种特殊的间隙锁，属于行级别锁。

MySQL如何加锁

加锁的对象是索引，加锁的基本单位是next-key lock，next-key lock是左开右闭区间。

在能使用记录锁或间隙锁就能避免幻读现象的场景下，next-key 就会退化成记录锁或间隙锁。

  1. 唯一索引等值查询：1. 记录不存在，找到第一条大于该查询记录的记录后，将该记录的索引中的next-key lock退化成间隙锁。

  2. 记录存在，定位到这一条记录，将该记录的索引中的next-key lock会退化为记录锁。

  3. 唯一索引范围查询：会对每一个扫描到的索引加next-key锁。

    1. 大于等于范围查询，存在等值查询条件，记录存在会退化成记录锁，之后的每一项都是next-key锁。
    2. 小于或小于等于范围查询
    • 条件值不在表中，扫描到终止范围查询的记录时，该记录的索引的next-key锁会退化成间隙锁。
    • 条件值在表中，如果是小于，扫描到终止范围的记录时，该记录的索引的next-key锁会退化为间隙锁。如果是小于等于，扫描到终止范围查询的记录时，该记录的索引next-key锁不会退化成间隙锁。

  4. 非唯一索引等值查询：会同时对主键索引和非唯一索引同时加锁，对主键加锁时，只有满足查询条件的记录才会对它们的主键索引加锁。

    • 查询记录存在，非唯一索引会存在索引值相同的记录，非唯一索引等值查询的过程是一个扫描的过程，直到扫描到第一个不符合条件的二级索引就停止扫描，扫描过程中加的是next-key锁，对于第一个不满足的记录该二级索引的next-key锁会退化为间隙锁。同时在符合查询条件的记录的主键索引上加记录锁。
    • 查询记录不存在，扫描到第一个不符合条件的二级索引，该二级索引next-key会退化为间隙锁。不存在满足查询条件的记录，不会对主键索引加锁。

  5. 非唯一索引范围查询：非唯一索引范围查询看，索引的next-key lock不会退化为间隙锁和记录锁的情况。

  6. 没有加索引的查询：如果查询语句没有走索引查询，导致扫描是全表扫描，每一条记录的索引上都会加next-key锁，相当于锁住了全表。

    在线上执行update、delete、select…for update等具有加锁性质的语句，一定要检查语句是否走了索引，如果是全表扫描，会对每一个索引加next-key锁，相当于锁住了整个表。

MySQL死锁

当有两个事务同时做查询不存在就插入的事务，且查询的值相同，当两个查询语句执行后，两个事务拥有相同的间隙锁，当执行插入操作时都在等待对方的事务释放间隙锁就会发生死锁。

Insert怎么加行级锁

Insert语句正常执行不会生成锁结构，靠聚簇索引记录自带的trx_id隐藏列来作为隐式锁来保护记录。

当事务需要加锁时，如果这个锁不可能发生冲突，InnoDB会跳过加锁环节，这种机制称为隐式锁。是一种延迟加锁机制。

如果插入数据时发生主键索引冲突，插入新记录的事务会给已存在的主键值重复的聚簇索引记录添加S型记录锁。

当第一个事务还未提交，有其它事务插入了与第一个事务相同的记录，第二个事务就会被阻塞，因此此时第一个事务插入的记录中的隐式锁会变为显式锁且类型是X型记录锁，而第二个事务是想对该记录加上S型next-key锁，X型与S型锁是冲突的。

日志

undo log（回滚日志）：是InnoDB存储引擎生成的日志，实现了事务中的原子性，主要用于事务回滚和MVCC。

redo log（重做日志）：是InnoDB存储引擎生成的日志，实现了事务中的持久性，主要用于掉电等故障恢复。

bin log（归档日志）：是server层生成的日志，主要用于数据备份和主从复制。

为什么需要undo log

undo log是一种用于撤销回退的日志，在事务未提交之前，MySQL会先记录更新前的数据到undo log日志文件里，当事务回滚时，可以利用undo log来进行回滚。

  • 插入一条记录时，把这条记录的主键值记下来。
  • 删除一条记录时，把这条记录的内容都记下来。
  • 更新一条记录时，把更新的列的旧值记下来。

一条记录的每一次更新操作产生的undo log格式都有一个roll_pointer指针和一个trx_id事务id：

  • 通过trx_id可以知道该记录是被哪个事务修改的；
  • 通过roll_pointer指针可以将这些undo log串成一个链表，这个链表就称为版本链。

undo log两大作用：

  • 实现事务回滚，保障事务的原子性。事务处理过程中，如果出现了错误或者用户执行了ROLLBACK语句，MySQL可以利用undo log中的历史数据将数据恢复到事务开始之前的状态。
  • 实现MVCC（多版本并发控制）关键因素之一。MVCC是通过ReadView + undo log实现的。undo log为每条记录保存多份历史数据，MySQL在执行快照读（普通select语句）的时候，会根据事务的Read View里的信息，顺着undo log的版本链找到满足其可见性的记录。

undo log和数据页的刷盘策略是一样的，都需要通过redo log保证持久化。buffer pool中有undo页，对undo页的修改也都会记录到redo log。redo log会每秒刷盘，提交事务时也会刷盘。

为什么需要buffer pool

MySQL启动时，InnoDB为buffer pool申请一片连续的内存空间，然后按照默认的16KB的大小划分出一个个页，buffer pool中的页就叫做缓存页。

buffer pool除了缓存索引页和数据页，还包括了undo页，插入缓存、自适应哈希索引、锁信息等。

为什么需要redo log

为了防止断电导致数据丢失的问题，当有一条记录需要更新的时候，InnoDB引擎就会先更新内存，然后将本次对这个页的修改以redo log的形式记录下来，这个时候就完成了。

InnoDB引擎会在适当的时候，由后台线程将缓存在buffer pool的脏页刷新到磁盘里，这就是WAL（Write-Ahead Logging）技术。

redo log是物理日志，记录了某个数据页做了什么修改，比如对xxx表空间中的xxx数据页xxx偏移量的地方做了xxx更新，每当执行一个事务就会产生这样的一条或多条物理日志。

当系统崩溃时，虽然脏页数据没有持久化，但是redo log已经持久化，接着MySQL重启后，可以根据redo log的内容，将数据恢复到最新状态。

redo log和undo log的区别：

  • redo log记录了此次事务完成后的数据状态，记录的是更新之后的值；
  • undo log记录了此次事务开始前的数据状态，记录的是更新之前的值；

事务提交之前发生崩溃，重启后会通过undo log回滚事务，事务提交之后发生崩溃，重启后会通过redo log恢复事务。

redo log要写磁盘，数据也要写磁盘，是否多此一举？

写入redo log的方式用了追加操作，所以磁盘操作是顺序写，而写入数据是随机写。磁盘的顺序写比随机写高效的多。

redo log的两大作用：

  • 实现事务的持久性，让MySQL有crash-safe的能力，保证MySQL在任何时间突然崩溃，重启之后已提交的记录都不会丢失。
  • 将写操作从随机写变成顺序写，提升MySQL写入磁盘的性能。

产生的redo log是直接写入磁盘的吗？

不是，redo log也有自己的缓存——redo log buffer。该buffer默认大小16KB，可以通过innodb_log_buffer_size参数调整大小，增大它的大小可以让MySQL处理大事务是不必写入磁盘，进而提升IO性能。

redo log什么时候刷盘？

刷盘时机：

  • MySQL正常关闭时；
  • 当redo log buffer中记录的写入量大于redo log buffer内存空间的一半时，会触发落盘；
  • InnoDB的后台线程每个1秒，将redo log buffer持久化到磁盘。
  • 每次事务提交时都将缓存在redo log buffer里的redo log持久化到磁盘。

innodb_flush_log_at_trx_commit参数控制：

  • 参数为0时，事务提交，不会主动触发写入磁盘的操作。通过调用write写到page cache，然后调用fsync持久化到磁盘。所以会导致上一秒内所有事务数据丢失。
  • 参数为1时，事务提交，直接持久化到磁盘。
  • 参数为2时，事务提交，写到redo log文件即page cache而不是直接写到磁盘。调用fsync，进程崩溃并不会丢失数据，只有在操作系统崩溃或系统断电才会丢失一秒的事务数据。

redo log文件写满了怎么办？

InnoDB存储引擎有1个重做日志文件组，有2个redo log文件组成，分别叫ib_logfile0和ib_logfile1。每个文件大小固定，上限1GB，总共2GB的操作。以循环写方式工作。用write pos标识写到的位置，用checkpoint表示当前要擦除的位置。如果write pos追上checkpoint意味着redo log文件满了，这时MySQL不能执行新的更新操作，会阻塞，此时会停下来将buffer pool中的脏页刷新到磁盘中。

为什么需要binlog

binlog文件是记录了所有数据库表结构变更和表数据修改的日志，不会记录查询类的操作，比如SELECT和SHOW操作。

redo log和binlog的区别

  1. 适用对象
    • binlog是server层实现，所有存储引擎可用
    • redo log是InnoDB存储引擎实现
  2. 文件格式
    • binlog有3中格式类型，分别是STATEMENT（默认格式）、ROW、MIXED。
      • STATEMENT：每一条修改数据的SQL都被记录到binlog中，主从复制中slave端再根据SQL语句重现，但又动态函数的问题。
      • ROW：记录行数据最终被修改称什么样了，没有动态函数问题，但每行数据的变化都会被记录，使binlog文件过大。
      • MIXED：包含上面两种模式，根据不同情况选择。
    • redo log是物理日志，记录在某个数据页做了什么修改。
  3. 写入方式
    • binlog追加写，写满一个文件就创建一个新的文件继续写，保存的是全量的日志。
    • redo log循环写，日志大小固定，保存未被刷入磁盘的脏页日志。
  4. 用途
    • binlog用于备份恢复、主从复制
    • redo log用于掉电等故障恢复

主从复制怎么实现？

MySQL的主从复制依赖于binlog，也就是记录MySQL上的所有变化并以二进制形式保存在磁盘上。复制的过程就是将binlog中的数据从主库传输到从库上。

主从复制过程：

  • 写入binlog：主库写binlog日志，提交事务，并更新本地存储数据。
  • 同步binlog：从库创建一个专门的IO线程连接主库的log dump线程，来接收主库的binlog日志，再把binlog信息写入relay log的中继日志里，再返回给主库复制成功的响应。
  • 回放binlog：从库会创建一个用于回放binlog的线程，去读relay log中继日志，然后回放到binlog更新存储引擎中的数据，最终实现主从的数据一致性。
主从复制模型
  • 同步复制：主库提交事务的线程要等待所有从库的复制成功响应才返回客户端结果。性能差：等所有返回响应，可用性差：任何库出问题都影响业务。
  • 异步复制：主库提交事务的线程并不等待binlog同步到各从库就返回客户端结果。宕机数据会丢失。
  • 半同步复制：只要等到任一从库响应就返回。即使出现主库宕机，至少有一个从库有最新的数据，不存在数据丢失的风险。

binlog什么时候刷盘？

事务执行过程中，先把日志写到binlog cache，事务提交时再把binlog cache写到binlog文件中。

一个事务的binlog不能被拆开，因此每个线程都分配了一片内存用于缓冲binlog，该内存叫binlog cache。事务提交时，执行器把binlog cache里的完整事务写入到binlog文件中，并清空binlog cache。虽然每个线程有自己的binlog cache，但最终都写到同一个binlog文件。

为什么需要两阶段提交？

事务提交后，redo log和binlog都要持久化到磁盘，但这是两个独立的逻辑，可能出现半成功的状态，会造成两份日志之间的逻辑不一致。

MySQL为了避免出现两份日志之间的逻辑不一致的问题，使用两阶段提交来解决，是分布式事务一致性协议。分为准备阶段和提交阶段，每个阶段都由协调者和参与者共同完成。

MySQL使用内部XA事务，由binlog作为协调者，存储引擎是参与者。当客户端执行commit语句或自动提交，MySQL内部开启XA事务，分两阶段事务的提交：

  • prepare阶段：将XID写入到redo log，同时将redo log对应的事务状态设置为prepare，然后将redo log持久化到磁盘；
  • commit阶段：把XID写入binlog，然后将binlog持久化到磁盘，接着调用引擎的提交事务接口，将redo log状态设置为commit，此时该状态并不需要持久化到磁盘，只需要write到文件系统的page cache就够了，因为只要binlog写磁盘成功，就算redo log的状态还是prepare也没关系，一样会被认为事务已经执行成功。

事务没提交的时候，redo log也是可能被持久化到磁盘的，但是binlog必须在事务提交之后才可以持久化到磁盘。

两阶段提交有什么问题？

保证了两个日志文件的数据一致性，但是性能很差，主要：

  • 磁盘IO次数高：对于双1配置，每个事务提交都会进行两次fsync
  • 锁竞争激烈：能保证单事务两个日志的内容一致，但在多事务情况下，却不能保证两者的提交顺序一致，因此在两阶段提交的流程上需要加一个锁来保证提交的原子性，从而保证多事务的情况下，两个日志的提交顺序一致。

MySQL引入了binlog组提交机制，当有多个事务提交的时候，会将多个binlog刷盘操作合并成一个，从而减少磁盘IO的次数。

引入了组提交机制后，prepare阶段不变，只针对commit阶段，将commit阶段拆分为三个过程：

  • flush阶段：多个事务按进入的顺序将binlog从cache写入文件；
  • sync阶段：对binlog文件做fsync操作；
  • commit阶段：各个事务按顺序做InnoDB commit操作。

上面的每个阶段都有一个队列，每个阶段有锁进行保护，因此保证了事务写入的顺序。且锁粒度减小了。

MySQL磁盘IO很高，有什么优化方法？

事务提交需要将binlog和redo log持久化到磁盘，可延迟binlog 和 redo log刷盘时机降低磁盘IO效率：

  • 设置组提交参数
  • 设置sync_binlog为大于1的N，表示事务提交都write但累计N个事务后才fsync。风险主机掉电会丢N个事务的binlog日志。
  • 将innodb_flush_log_at_trx_commit设置为2，表示redo log写到redo log文件，但不是持久到磁盘。风险主机掉电会丢数据。

Redis常见面试题

什么是Redis

一种基于内存的数据库，对数据的读写都是在内存中完成，读写速度非常快，常用于缓存、消息队列、分布式锁等场景。提供了多种书类型来支持不同的业务场景。

为什么Redis作为MySQL的缓存

主要因为Redis具备高性能（数据在内存中，操作速度快）和高并发（QPS非常高）两种特性。

Redis数据类型以及使用场景

常见五种：string、hash、list、set、zset。后面加了四种：Bitmap（2.2加）、HyperLogLog（2.8加）、GEO（3.2加）、Stream（5.0加）。

  • String类型：缓存对象，常规计数、分布式锁、共享session信息等。
  • List类型：消息队列（有两个问题：生产者需要自行实现全局唯一ID；不能以消费组形式消费数据）等。
  • Hash类型：缓存对象，购物车等。
  • Set类型：聚合计算（并交差），比如点赞、共同关注、抽奖活动等。
  • Zset类型：排序场景，比如排行榜、电话和姓名排序等。
  • BitMap类型：二值状态统计的场景，比如签到、判断用户登录状态、连续签到用户总数等；
  • HyperLogLog类型：海量数据基数统计的场景，比如百万级网页UV计数等。
  • GEO类型：存储地理位置信息的场景，比如滴滴叫车。
  • Stream类型：消息队列，相比于List有两个特性：自动生成全局唯一消息ID，支持以消费组形式消费数据。

数据类型底层实现

  • string类型：底层数据结构主要是int和SDS。SDS相较于C中的字符串不仅可以保存文本数据还可保存二进制数据，因为不以'\0'判断结束。获取字符串长度时间复杂度为O(1)，且SDS API是安全的，不会缓冲区溢出。

    字符串对象的内部编码有3种：int、raw和embstr。
    如果字符串对象保存的是整数值，并且可用long表示就直接保存在字符串对象的ptr中。如果字符串对象保存的是字符串且长度小于等于32字节（redis3、4是39字节，redis5是44字节），则字符串对象和SDS分配在连续空间中，字符串保存在SDS中的buf。大于则redisObject和SDS两次分配。

  • List类型：最长2^32^ - 1，即超过40亿。redis3.2之前元素小于512个，每个元素小于64字节会使用压缩列表；不满足则使用双向链表。redis3.2之后只使用quicklist实现。

    应用场景：消息队列（需求：消息保序（list本身的先进先出保证）、处理重复的消息（手动为每个消息生成成一个全局唯一ID并包含在消息中）和保证消息可靠性（BRPOPLPUSH命令取出消息然后再插入）。缺陷：不支持多个消费消费同一条消息，不支持消费组的实现）

  • Hash类型：元素个数小于512，每个元素小于64字节会使用压缩列表；不满足使用哈希表。Redis7.0之后，使用listpack和哈希表实现。

    应用场景：hash类型的（key，field，value）的结构与对象（对象id，属性，值）的结构相似可存储对象（string+Json也是存储对象的一种方式）；购物车亿以用户id为key，商品id为field，商品数量为value。

  • Set类型：最多存储2^32^-1个元素，支持交、并、差等集合操作，是无序的且非重复元素。元素小于512时使用整数集合实现，不满足使用哈希表实现。

    应用场景：用来数据去重。set的交并差计算复杂度较高，数据量大时可能会导致Redis阻塞，可选择从库上完成。点赞：保证每个用户只能点一个赞，key是文章id，member是用户id；共同关注：交集运算可用来计算共同关注的好友、公众号等，key是用户id，value是已关注的公众号；抽奖活动：key为抽奖活动名，value为员工名称，如果允许重复使用SRANDMEMBER随机选出，不重复使用SPOP命令。

  • Zset类型：比Set类型多一个排序属性score且有序。元素个数小于128个，每个元素值小于64字节使用压缩列表实现，不满足使用跳表实现。Redis7.0后，压缩列表由listpack替代。

    Zset结构体有两个数据结构，一个跳表，一个哈希表。这样的好处是既能进行高效的范围查询，也能进行高效单点查询。

    应用场景：排行榜：用户id为key，文章为member，点赞数为score，会按score升序排列；

  • BitMap类型：位图是一系列二进制数组，可以通过偏移量定位元素。适合数据量大且二值统计的场景。用string类型作为底层实现来保存二进制数组。

    应用场景：签到统计：每一个bit表示用户每一天的签到；判断用户登录状态：用户id作为offset，value表示是否用户登录；连续签到用户总数：日期作为key，userid作为offset，打卡value为1

  • HyperLogLog类型：Redis2.8.9新增，用于统计基数的数据集合类型，基数统计就是指统计一个集合中不重复的元素个数，统计规则是基于概率完成的不准确。优点是输入元素数量或体积非常大时计算基数所需的内存空间总是固定的。每个HyperLogLog键只需要花费12KB内存，就可以计算接近2^64^个不同元素的基数。内部实现设计很多数学问题

    应用场景：百万级网页UV计数。

  • GEO类型：Redis3.2新增，用于存储地理位置信息。直接使用ZSET实现，GEO使用GeoHash编码方法实现了经纬度到ZSET中分值的转换，两个关键机制就是【对二维地图区间划分】和【区间编码】。一组经纬度在某个区间后，就用区间的编码值来表示，并把编码值作为分值。

    应用场景：滴滴打车，GEOADD cars:locatios 116.034579 39.030452 33把id为33的车辆的当前经纬度位置存入GEO集合中，GEORADIUS cars:locations 116.054579 39.030452 km ASC COUNT 10根据输入的经纬度查找以这个经纬度为中心的5公里内的车辆信息。

  • Stream类型：Redis5.0新增，专门为消息队列设计。

    应用场景：消息队列，stream使用内部队列留存消费组里每个消费者读取的消息直到消费者使用XACK命令通知stream消息已处理完成。

Redis数据结构

SDS

扩容方式，如果所需sds小于1MB就2倍扩容，超过1MB就扩容字符串所需长度外加1MB空间

就Redis中flags成员变量表示SDS类型，主要区别是数据结构中的len和alloc成员变量的数据类型不同。采用__attribute__((packed))属性定义结构体避免内存对齐。

链表

链表每个节点内存不连续，无法很好利用CPU缓存。链表节点的值都需要一个链表节点结构头分配，内存开销大。

压缩列表

内存紧凑、连续内存空间，针对不同长度的数据进行相应编码。

缺点：不能保存过多的元素否则查询效率低；新增或修改时需重新分配内存，可能引发连锁更新的问题。

prevlen和encoding根据数据的大小和类型来进行不同的空间的大小分配。如果前一个字节的长度小于254字节，prevlen需要1字节空间保存。大于等于254字节prevlen需要5字节空间保存。encoding属性的空间大小跟数据还是字符串还是整数，以及字符串长度有关。如果是整数，encoding会使用1字节空间编码，如果是字符串，根据字符串大小会使用1字节/2字节/5字节空间进行编码。

压缩列表新增或修改某个元素，空间不够就需要重新分配。新插入的元素较大时会导致后续元素的prevlen占用空间都发生变化，从而引起连锁更新，导致每个元素的空间都要重新分配，造成访问压缩列表性能下降。

哈希表

将key通过hash函数的就能定位数据在表中的位置，因为哈希表实际上是数组。

rehash：用到了两个哈希表，正常服务请求阶段都会写入到哈希表1，此时哈希表2并没有倍分配空间。数据增多触发rehash操作，过程分3步：

  • 给哈希表2分配空间，一般比哈希表1大2倍；
  • 将哈希表1的数据迁移到哈希表2中
  • 迁移完成后，哈希表1的空间会被释放，并把哈希表2设置为哈希表1，然后在哈希表2新创建一个空白的哈希表，为下次rehash做准备。

渐进式rehash：迁移工作不是一次性完成，而是分多次。渐进式rehash步骤：

  • 给哈希表2分配空间；在rehash进行期间，每次哈希表元素进行新增、删除、查找或更新操作时，Redis除了会执行对应的操作之外，还会顺序将哈希表1中的索引位置上的所有key-value迁移到哈希表2上。
  • 随着处理客户端哈希表请求数量越多，最终在某个时间点会把哈希表1的所有key-value迁移到哈希表2从而完成rehash操作。

rehash触发条件：负载因子 = 哈希表已保存节点数量 / 哈希表大小。

  • 负载因子大于等于1，并且Redis没有执行bgsave命令或bgrewriteaof命令，也就是没有执行RDB快照或没有进行AOF重写时，就会rehash操作。
  • 负载因子大于等于5，哈希冲突非常严重，不管有没有执行RDB或AOF重写都强制rehash操作。
整数集合

content声明为int8_t类型，但是实际保存取决于encoding的值，如果INTSET_ENC_INT16，那么content就是int16_t类型，如果INTSET_ENC_INT32那content就是int32_t，如果是INTSET_ENC_INT64，那content就是int64_t。

整数升级，但插入一个长度更长的类型，会先进行扩容，将短的整数都变为长的类型并移到对应的空间。

不支持降级操作。

跳表

在链表的基础上改进实现了一种多层的有序链表。

跳表节点层数设置：相邻两层的节点数量理想比例是2：1.查找复杂度可以降到O(logN)。采用在创建节点时，随机生成每个节点的层数，不严格维持2：1。跳表创建时生成范围【0-1】的一个随机数，如果这个随机数小于0.25那么层数就增加1层，继续生成下一个随机数直到随机数结果大于0.25结束，最终确定该节点的层数。层高最大限制是64.

使用跳表而不使用平衡树的原因：

  • 内存占用：跳表比平衡树更灵活，可以改变增加一层的概率值从而改变内存占用情况
  • 范围查找：跳表比平衡树更简单，平衡树需要中序遍历。
  • 实现难度：跳表比平衡树更简单，平衡树的插入和删除可能引发子树调整，而跳表的插入和删除只需要修改相邻节点的指针。
quicklist

是双向链表和压缩列表的组合，通过控制每个链表节点中的压缩列表的大小或元素个数，来规避连锁更新的问题。添加元素时不会直接新建一个节点，而是会检查插入位置的压缩列表能否容纳，不能容纳才会新建一个新的节点。

listpack

设计目的是替代压缩列表，最大特点是listpack不包含前一个节点的长度。还是用一块连续内存空间紧凑保存数据，并且采用不同的编码方式保存不同大小的数据。listpack没有压缩列表中记录前一个字节长度的字段了，listpack只记录当前节点的长度。避免了连锁更新问题。因为len保存在每个节点空间的最后，可以向左解析得到上一个字节的长度。

Redis线程模型

单线程——接收客户端请求->解析请求->数据读写->发送客户端 这些过程是由一个线程完成的

  • Redis2.6——会起两个后台线程，分别处理关闭文件、AOF刷盘
  • Redis4.0以后——增加一个后台线程，用来异步释放Redis内存也就是lazyfree线程。例如执行unlink key/flushdb async/flushall async等命令会把删除操作交给后台线程。因此删除大key时不要用del命令，因为del是在主线程处理的。

使用单线程的原因：操作基本都在内存完成，采用了高效的数据结构，因此Redis瓶颈出现在机器内存或网络带宽，而非CPU。还可以避免多线程之间的竞争，省去线程切换开销，不会出现死锁。

多线程——Redis6.0之后采用多个IO线程来处理网络请求，因为随着网络硬件性能提升导致性能瓶颈可能出现在网络IO处理上。但是对于命令的执行还是使用单线程，且默认情况IO多线程只针对发送响应数据。要多线程处理读请求需要修改配置io-threads-do-reads为yes。

Redis持久化

AOF日志

Redis执行完一条写操作命令后就把该命令以追加的方式写入到一个文件中，Redis重启时会读取该文件记录的命令来数据恢复。日志内容是协议。

先执行命令、再写入日志——避免因语法错误而未执行却记录到AOF日志中，会导致恢复出错。不会阻塞当前写操作命令执行。但会有数据丢失的风险，但AOF日志也是在主线程执行会阻塞后续操作。

AOF写回策略

appendfsync配置项：

  • Always：每次写操作执行完后同步AOF日志到硬盘
  • Everysec：每次写操作执行完先将命令写入AOF文件内核缓冲区，每隔一秒写到磁盘
  • No：每次写操作执行完先将命令写入到AOF文件内核缓冲区，由操作系统写到磁盘
AOF重写机制

AOF日志文件过大会带来性能问题，比如重启恢复时会很慢。文件大小超过设定阈值就启用重写机制。

重写时，读取当前数据库中的所有键值对，然后将每一个键值对用一条命令记录到新的AOF文件，记录完后将新的AOF文件替换掉现有的AOF文件。

重写AOF过程是由后台子进程bgrewriteaof完成——避免阻塞主进程；子进程带有主进程的数据副本。

重写时主进程执行的命令子进程无法知道——Redis设置AOF重写缓冲区，执行完写操作将命令写入【AOF缓冲区】和【AOF重写缓冲区】。

当子进程完成AOF重写工作，会将主进程发送一条信号，主进程收到信号会调用信号处理函数，该函数负责——将AOF重写缓冲区所有内容追加到新的AOF文件中；将新的AOF文件改名覆盖原AOF文件——这是在主进程中执行，会阻塞。

RDB快照

AOF记录操作命令不是实际数据，恢复需要把日志命令都执行一遍，恢复会慢。

RDB快照是记录某个瞬间的内存数据，记录实际数据。RDB恢复数据效率比AOF高很多。

Redis提供两个命令生成RDB文件：

  • save命令：主线程执行，会阻塞
  • bgsave命令：创建一个子进程生成RDB文件，避免主线程阻塞。

Redis可以通过配置文件的选项来实现每隔一段时间自动执行一次bgsave命令。Redis是全量快照，执行快照是比较重的操作，太频繁会对Redis性能产生影响。频率低可能导致数据丢失太多。

混合持久化

RDB数据恢复快，频率不好把握；AOF丢失数据少，但数据恢复慢。

Redis4.0提出混合使用AOF日志和内存快照。

混合持久化工作在AOF日志重写过程，AOF重写时，重写子进程回先将内存数据以RDB方式写入AOF文件，然后将重写缓冲区里的增量命令以AOF方式写入，写入完成通知主进程将新的RDB格式和AOF格式的AOF文件替换旧的AOF文件。——这里有个问题，重写缓冲区子进程应该访问不到吧，将重写缓冲区内容写到AOF文件应该是主进程的任务吧！！！

优点：开头RDB格式加载很快，同时有AOF优点，降低大量数据丢失风险。
缺点：AOF文件中有RDB格式可读性差；只适用于Redis4.0之后版本，有兼容性问题。

Redis集群

Redis实现服务高可用

从Redis的多服务节点考虑，比如Redis的主从复制、哨兵模式、切片集群。

主从复制

主从复制是Redis高可用服务的最基础保证，将一台主Redis服务器同步数据到多态从Redis服务器，即一主多从模式，且主从服务器之间采用读写分离方式——主服务器可以进行读写操作，从服务器一般是只读并接收执行主服务器同步过来的写操作。

主从服务器之间的命令复制是**异步进行的。也就是主服务器在本地执行完不等从服务器也执行完就直接返回给客户端结果。可能会导致主从服务器间数据不一致，无法实现强一致性保证**。

哨兵模式

使用Redis主从服务时如果出现故障宕机需要手动恢复，为解决这，增加了哨兵模式，监控主从服务器，并提供主从节点故障转移功能。

切片集群模式

当Redis缓存数据量大到一台服务器无法缓存时就需要使用Redis切片集群方案，将数据分布在不同的服务器上，一次降低系统对单节点的依赖，提高Redis服务的读写性能。

Redis Cluster采用哈希槽来处理数据和节点之间的映射关系。一个切片集群共有16384个哈希槽，每个键值对根据它的key被映射到一个哈希槽中具体执行过程：

  • 根据键值对的key，按照CRC16算法计算出一个16bit的值。
  • 再用16bit值对16384取模，得到0~16383范围内的模数，就是对应的哈希槽编号。

哈希槽映射到具体的Redis节点有两种方案：

  • 平均分配：Redis会自动把所有哈希槽平均分布到集群节点上。
  • 手动分配：可以使用cluster meet命令手动建立节点间的连接，组成集群，再使用cluster addslots命令指定每个节点上的哈希槽个数。

手动分配哈希槽时，需要把16384个槽都分配完，否则Redis集群无法正常工作。

集群脑裂导致数据丢失

主节点与所有从节点都失联，但与客户端网络正常。所有数据都在主节点中，无法同步给从节点。哨兵发现主节点失联会在所有从节点中选举一个leader作为主节点，这时集群就有两个主机点——脑裂出现了。

网络好了哨兵会将旧主节点降级为从节点，想新主节点请求数据同步，此时旧主节点会清空自己本地数据，会导致之前客户端在旧主节点中写入到数据丢失，也就是集群产生脑裂数据丢失问题。

解决方案：当主节点发现从节点下线或者通信超时的总数量小于阈值时，旧禁止主节点进行写数据，直接把错误返回给客户端。

在Redis配置文件中有两个参数可设置：

  • min-slaves-to-write x：主节点必须要有x个从节点连接，否则主节点禁止写数据
  • min-slaves-max-lag x：主从数据复制和同步的延迟超过x秒回禁止写数据

Redis过期删除与内存淘汰

过期删除策略

Redis可以对key设置过期时间。会把该key带上过期时间存储到一个过期字典中。当查询一个key时，Redis首先检查该key是否存在于过期字典中。

Redis使用的过期策略是惰性删除+定期删除配合使用：

  • 惰性删除：不主动删除过期键，每次范文key时检查是否过期，过期就删除该key。
    优点：使用很少的系统资源，对CPU时间友好
    缺点：如果一个过期key一直不访问就无法释放。

  • 定期删除：每个一段时间随机从数据库中取出一定数量的key进行检查，并删除其中的过期key。

    定期删除过程：从过期字典中随机抽20个key，删除其中过期的key，如果过期key超过25%占比就再抽20个key。为防止一直循环设置了执行是时间上限默认25ms。

    优点：限制删除操作时长和频率减少CPU占用，减少了过期键对空间的无效占用
    缺点：难以确定执行时长和频率。

Redis持久化对过期键的处理

RDB快照：

  • RDB文件生成：会对key进行过期检查，过期的key不会保存在RDB文件中。
  • RDB加载：
    • 主节点：程序会对文件中保存的key进行检查，过期key不会被载入。
    • 从节点：不检查直接载入。

AOF持久化：

  • AOF文件写入：过期键没删除回保留过期key，当删除了回追加一条DEL命令。
  • AOF重写：会进行过期key检查，已过期的不会被保存到重写后的AOF文件中。

主从模式过期键处理

  • 从库不进行过期扫描，对过期的处理是被动的。
  • 从库过期键处理依靠主服务器，主库key到期会在AOF文件里增加一条del命令，同步到所有的从库。

内存淘汰策略

Redis运行内存达到了某个阈值，会触发内存淘汰机制，配置文件中的maxmemory.
